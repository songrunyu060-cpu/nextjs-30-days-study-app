## 案例代码 ①：Middleware —— 保护 `/users`（示意）

在仓库根目录新建 `middleware.ts`（**学习分支**试用；当前主线未包含）：

```ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  // 示意：从 cookie 读 session（接入 Auth 后替换为真实校验）
  const token = request.cookies.get("session")?.value;
  if (!token && request.nextUrl.pathname.startsWith("/users")) {
    return NextResponse.redirect(new URL("/login", request.url));
  }
  return NextResponse.next();
}

export const config = {
  matcher: ["/users/:path*"],
};
```

**注意**：`middleware` 在 Edge 运行，**不要**在此直连 Neon 做重查询；复杂鉴权可解码 JWT 后把 `userId` 放进 `request.headers` 供 Server Component 使用（模式因项目而异）。

## 案例代码 ③：CSRF 与 Server Actions

Next 对 Server Actions 内置请求校验；**仍需**：

- 敏感操作使用 **POST**（你已有 `<form>` + `deleteUserFormAction`）；
- **不要**把「是否管理员」仅放在 `hidden` 字段，必须在服务端根据 **Session** 判断。

---

## 1. Middleware：适合做什么

适合：

- **重定向**：未登录去 `/login`；已登录禁止访问 `/login`。
- **Header 注入**：请求 ID、CSP（需与部署平台配合）。
- **地理/AB**：轻量分流。

不适合 / 需谨慎：

- **重型逻辑、长连接数据库**：Edge 限制与冷启动；复杂鉴权可放到 **Server Component / Action** 内。

**Neon**：Middleware 内若查库，要评估 **延迟与连接模型**；通常更推荐 **Session 解码在 Middleware，业务授权在服务端**。

---

## 2. Server Actions：必须当「API」对待

每条 Action 应明确：

1. **谁可以调用**（已登录？管理员？资源所有者？）
2. **入参是否可信**（`FormData` 可被篡改；**永远不要**仅凭 `hidden` 的 `userId` 信任身份）
3. **副作用范围**（删库前二次确认、幂等）

**对本项目**：`deleteUserFormAction`、`saveUserAction` 在 production 前应加 **会话校验** 与 **角色检查**。

---

## 3. CSRF 与 Server Actions

Next 对 Server Actions 有 **内置防护机制**（同源与 token 等策略随版本演进）。实践建议：

- **仍**避免把敏感操作暴露给「可被任意站点提交的表单」场景；
- 使用 **POST** 语义的操作走 Action；**GET 只做幂等读**（你的删除已用 form POST，方向正确）。

具体细节以 **当前 Next 官方 Security 文档** 为准。
