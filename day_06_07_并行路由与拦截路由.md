# Day 6-7：路由高级技巧 — 并行路由（Parallel Routes）与拦截路由（Intercepting Routes）

## 学习目标

- 理解 **Parallel Routes**：用 **`@slot`** 在同一布局中并行渲染多个页面槽位（如主内容 + 侧栏 + Modal 槽）。
- 理解 **Intercepting Routes**：用 **`(.)`** 等约定在**软导航**时拦截子路由，渲染「弹窗」；**硬刷新/直链**仍打开全页。
- 能将本项目的 **`/users?edit=` / `?create=`** 模式，与「Modal 作为独立段」的架构做对比与迁移规划。

---

## 与本项目当前实现的关系

当前仓库中，用户相关 UI 大致为：

- **列表页**：`src/app/(main)/users/page.tsx`
- **编辑/新建**：通过 **`searchParams`**（`edit`、`create`）控制 **`UserModal`** 是否打开（可分享 URL，但**不是**拦截路由段）。

这是一种**有效且简单**的做法。Day 6-7 的路由技巧是在此之上的 **UX 与信息架构升级**，不是「必须立刻重写」。

## 案例代码 ①：本项目 —— 用 `searchParams` + `Link` 驱动 Modal

**页面读取参数并决定是否渲染弹窗**（`src/app/(main)/users/page.tsx` 核心逻辑）：

```tsx
const sp = await searchParams;
const q = (sp.q ?? "").trim();
const editId = /* 从 sp.edit 解析 number */;
const editingUser = Number.isFinite(editId)
  ? allUsers.find((u) => u.id === editId)
  : undefined;
const createOpen =
  sp.create !== undefined && String(sp.create).trim() !== "";

{editingUser && (
  <UserModal
    editingUser={editingUser}
    open
    closeHref={q !== "" ? `/users?q=${encodeURIComponent(q)}` : "/users"}
  />
)}
{createOpen && (
  <UserModal open closeHref={q !== "" ? `/users?q=${encodeURIComponent(q)}` : "/users"} />
)}
```

**列表行「编辑」链到带 `edit` 的 URL**（`UserList.tsx`）：

```tsx
<Link href={`/users?edit=${user.id}${qParam}`}>编辑</Link>
```

**关闭弹窗 = 回到无 query 的列表**（`UserModal.tsx` 内 `DialogClose` + `Link`）：

```tsx
<DialogClose asChild>
  <Link href={href} aria-label="关闭">×</Link>
</DialogClose>
```

## 案例代码 ②：并行路由 —— `layout` 接收 `@modal` 槽（学习用 POC）

在**新分支**可尝试如下结构（与当前 `users` 并存，不必替换）：

```tsx
// app/(main)/layout.tsx
export default function MainLayout({
  children,
  modal,
}: {
  children: React.ReactNode;
  modal: React.ReactNode;
}) {
  return (
    <>
      {children}
      {modal}
    </>
  );
}
```

```tsx
// app/(main)/@modal/default.tsx
export default function DefaultModal() {
  return null;
}
```

```tsx
// app/(main)/users/@modal/(.)users/[id]/page.tsx —— 拦截段示意（路径以官方约定为准）
export default function UserModalPage() {
  return <UserModal open /* … */ />;
}
```

具体 `(.)`、`(..)` 段名需对照 [Intercepting Routes](https://nextjs.org/docs/app/building-your-application/routing/intercepting-routes) 按目录层级微调。

---

## 1. Parallel Routes（`@folder`）

### 1.1 解决什么问题

同一 URL 布局下需要 **多个区域独立加载、独立失败、独立 Loading**。例如：

- `@main`：主内容
- `@modal`：模态槽（可为空或渲染子页面）

### 1.2 典型目录结构（示意）

```
app/
  layout.tsx          # props.children + props.modal
  @modal/
    default.tsx       # 无 Modal 时占位
  (main)/
    users/
      page.tsx
```

### 1.3 要点

- 根 `layout` 接收 **具名插槽** `modal`（与 `@modal` 文件夹名对应）。
- **`default.tsx`**：当子路由不激活该槽时渲染什么（避免 404）。

---

## 2. Intercepting Routes（`(.)segment`）

### 2.1 解决什么问题

用户从列表点进「详情」时希望 **弹窗覆盖**；复制链接新开或刷新时希望 **整页详情**。  
拦截路由用 **路由段** 表达 Modal 内容，配合 Parallel Routes 的 `@modal` 槽实现。

### 2.2 约定（简化）

- `(.)`：同级拦截  
- `(..)`：上一级  
- 具体规则以 [Next.js 官方文档 - Intercepting Routes](https://nextjs.org/docs/app/building-your-application/routing/intercepting-routes) 为准。

### 2.3 与 `Link` / `prefetch`

你项目 `NavLinks.tsx` 使用 **`prefetch={true}`**。拦截路由场景下，预取行为会影响「是否提前拉到拦截段」——出现异常时要结合 **Network** 与 **软/硬导航** 排查。

---

## 3. 与本项目 `UserModal` 的对比

| 维度 | 当前 `searchParams` + `UserModal` | Parallel + Intercept |
|------|-----------------------------------|------------------------|
| URL 语义 | `?edit=1` 查询参数 | 常为 `/users/123` 独立段 |
| 直链与刷新 | 行为一致（同一页） | 可区分「弹窗」与「全页」 |
| 实现复杂度 | 低 | 高（槽位、`default`、心智负担） |
| SEO | 列表页为主，参数可索引性需自行设计 | 详情段 URL 更清晰 |

**建议**：学习阶段在 **分支** 上新建 `app/(main)/users/@modal/...` 做 POC，验证团队是否接受该复杂度。

---

## 4. 与 `next.config.ts` 的 `staleTimes`

软导航时 **Router Cache** 会缓存 RSC 结果一段时间。Modal 拦截 + 并行槽位下，用户会感觉「返回时列表是否陈旧」——与 **Day 8-9 缓存**、**`router.refresh()`** 同属「体感一致性」问题。

## 6. 小结

- **Parallel Routes**：多区域并行；适合「主内容 + 侧栏/Modal 槽」。
- **Intercepting Routes**：软导航弹窗、硬导航整页；适合电商图集、用户资料卡等。
- **本项目**用 `searchParams` 已达成可分享 Modal 状态；进阶再考虑 `@modal` 架构。
