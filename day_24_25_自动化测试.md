# Day 24-25：自动化测试 — Vitest（Service）与 Playwright（Server Actions 流程）

## 学习目标

- 用 **Vitest** 覆盖 **`features/*/use-cases.ts`、`user.service.ts`** 等纯逻辑与数据适配层（可 mock DB）。
- 用 **Playwright** 做 **E2E**：从页面打开 `/users` → 新建/编辑/删除 → 断言 UI 与 URL。
- 明确：**本仓库 `package.json` 当前未包含测试脚本与依赖**，以下内容为**落地清单**。

---

## 与本项目结构

| 测试类型 | 建议目标路径 |
|----------|----------------|
| 单元 / 集成 | `src/features/user/use-cases.test.ts`、`user.service.test.ts` |
| E2E | `e2e/users.spec.ts`（目录自定） |

**Neon**：测试环境应使用 **独立数据库** 或 **Testcontainers / 本地 Postgres**，**切勿**对生产串执行破坏性用例。

## 案例代码 ①：Vitest —— 测 `saveUserUseCase` 校验分支

`src/features/user/use-cases.test.ts`（新建；需先 `pnpm add -D vitest`）：

```ts
import { describe, it, expect, vi } from "vitest";
import { saveUserUseCase } from "./use-cases";

vi.mock("@/features/user/user.service", () => ({
  saveUserService: vi.fn(async () => []),
}));

describe("saveUserUseCase", () => {
  it("空对象应抛 VALIDATION_FAILED", async () => {
    await expect(saveUserUseCase(null)).rejects.toThrow("VALIDATION_FAILED");
  });

  it("缺少 email 应失败", async () => {
    await expect(
      saveUserUseCase({ name: "张三" }),
    ).rejects.toThrow("VALIDATION_FAILED");
  });
});
```

`package.json` 增加脚本：`"test": "vitest"`。

## 案例代码 ②：Playwright —— 打开 `/users` 并断言标题

`e2e/users.spec.ts`：

```ts
import { test, expect } from "@playwright/test";

test("用户页标题可见", async ({ page }) => {
  await page.goto("/users");
  await expect(
    page.getByRole("heading", { name: "用户管理系统" }),
  ).toBeVisible();
});
```

`playwright.config.ts`（节选）：

```ts
import { defineConfig } from "@playwright/test";

export default defineConfig({
  use: { baseURL: "http://127.0.0.1:3000" },
  webServer: {
    command: "pnpm dev",
    url: "http://127.0.0.1:3000",
    reuseExistingServer: !process.env.CI,
  },
});
```

## 1. Vitest 配置要点

1. 安装：`pnpm add -D vitest @vitejs/plugin-react`（按需）。
2. `vitest.config.ts` 中 `resolve.alias` 对齐 **`@/`**（与 `tsconfig` paths 一致）。
3. **Mock `db`**：对 `saveUserUseCase` 测校验逻辑时，可 mock `@/db` 避免真连 Neon。

**优先测**：`userCreateSchema` / `userUpdateSchema` 边界、`omitEmptyStrings` 行为、`VALIDATION_FAILED` 分支。

---

## 2. Playwright 配置要点

1. 安装：`pnpm add -D @playwright/test` 并 `pnpm exec playwright install`。
2. `playwright.config.ts`：`baseURL: 'http://127.0.0.1:3000'`，`webServer` 启动 `pnpm dev` 或 `pnpm start`。
3. **Server Actions** 无独立 REST 路径，E2E 通过 **UI 与 URL** 断言（与真实用户一致）。

**示例场景**（描述级）：

- 访问 `/users?q=test` → 表格行数变化；
- 点击「新增用户」→ URL 含 `create` → 提交 → `revalidate` 后列表出现新行。

---

## 3. CI 建议

- **GitHub Actions**：`pnpm install` → `pnpm lint` → `pnpm test` → `pnpm build`；E2E 可仅在 `main` 跑或夜间跑。
- **Vercel**：构建阶段通常不连真实 E2E；E2E 放在 CI。

---

## 4. 与 Drizzle Migrate

- CI 前对测试库跑 **`pnpm db:migrate`**（指向测试 `DATABASE_URL`）。
