## 0. 先看你项目当前状态（真实代码）

你现在的关键链路是：

1. 页面渲染时读取用户：`src/app/(main)/users/page.tsx` 里调用 `getAllUsers()`
2. 写操作走 Server Action：`src/features/user/user.action.ts`
3. 写完后调用 `revalidateTag("users", "fastCache")`（必要时再 `refresh()` 让当前用户立刻刷新 UI）

这意味着你现在采用的是：

- **读**：DB 直查（Drizzle）但已接入 **Data Cache**（`"use cache"` + `cacheTag("users")` + `cacheLife("fastCache")`）
- **写**：通过 **`revalidateTag` 按业务 tag 精准失效**，并可用 `refresh()` 解决“当前页面不立刻更新”的体验问题

这套模式在 CRUD 场景是成立的：既能让**后续所有请求**拿到最新数据（服务端缓存已失效），也能在需要时让**当前操作者**立刻看到最新列表（`refresh()`）。

## 案例代码摘录：本项目「读库 + 写后 revalidate」

**读（`"use cache"`：函数级 Data Cache + tag + profile）** — `src/features/user/user.service.ts`：

```ts
export async function getAllUsers(): Promise<User[]> {
  "use cache";
  cacheTag("users");
  cacheLife("fastCache");
  return await db.select().from(users).orderBy(desc(users.id));
}
```

**写（Server Action 末尾失效 `users` tag；必要时刷新当前客户端）** — `src/features/user/user.action.ts`：

```ts
import { revalidateTag, refresh } from "next/cache";

function revalidateUserViews() {
  revalidateTag("users", "fastCache");
  // refresh() 只影响“触发 action 的当前客户端”，用于让 UI 立刻更新
  refresh();
}

export async function deleteUserFormAction(formData: FormData) {
  const id = Number(formData.get("id"));
  if (!Number.isFinite(id)) return;
  await deleteUser(id);
  revalidateUserViews();
}
```

> 备注：你已经在用 `"use cache" + cacheTag/cacheLife`，不必再为了“能 tag 失效”去强行换成 `unstable_cache`；二者是同一条“Data Cache 管线”的不同入口（学习目的可对比两种写法）。

## 1. 四个概念先“对齐语义”

### 1.1 `force-cache`

- 作用对象：主要是 `fetch`
- 含义：把这次请求结果尽量当作可缓存内容复用（静态倾向）
- 典型形式：

```ts
await fetch(url, { cache: "force-cache" });
```

### 1.2 `revalidate`（时间维度）

- 作用对象：`fetch` 或路由段配置
- 含义：缓存内容过 `N` 秒后视为“过期可重取”（ISR 思路）
- 典型形式：

```ts
await fetch(url, { next: { revalidate: 60 } });
```

### 1.3 `unstable_cache`（函数级缓存）

- 作用对象：任意异步函数（例如直接查 DB 的函数）
- 含义：把函数返回值放进 Data Cache，可配 `revalidate` 与 `tags`
- 典型形式：

```ts
const getCached = unstable_cache(fn, ["key"], { revalidate: 60, tags: ["x"] });
```

### 1.4 `tags` + `revalidateTag`（事件驱动失效）

- 作用对象：被打了同一 `tag` 的缓存条目
- 含义：写操作完成后，按业务实体批量清理缓存（比路径更“语义化”）
- 典型形式：

```ts
// Next 16.2.0：revalidateTag 需要传 (tag, profile)
revalidateTag("users", "fastCache");
```

## 1.5 Next.js 四层缓存全景（结合 Drizzle 的直观版）

下面把「请求内去重 → 数据缓存 → 路由级产物 → 客户端路由」和**你项目里的 Drizzle 查询**对齐。**`unstable_cache` 落在第二层：Data Cache（数据缓存）**：它和带缓存选项的 `fetch` 同属 **Data Cache**，而不是请求级 memo。

| 层级 | 典型 API / 机制 | 生命周期与作用 | 在你项目中的落点（示意） |
|------|----------------|----------------|--------------------------|
| **1. Request Memoization（请求内去重）** | React `cache()`；同一请求里相同 `fetch` 也会自动去重 | **仅在一次服务端渲染 / RSC 请求**内复用；**新请求会重新执行** | 用 `cache(async () => db…)` 包 `getAllUsers`：**同一页、同一请求**里多处调用只查库一次；**不**相当于跨刷新持久化 |
| **2. Data Cache（数据缓存）** | `fetch` + `cache` / `next: { revalidate, tags }`、`unstable_cache` | **可跨多次请求复用**（服务器侧 Data Cache）；用 `revalidateTag`、`revalidate` 或 `revalidatePath` 等失效 | **Drizzle 直查进这一层，就靠 `unstable_cache`**（或未来的等价 API）；这是「非 HTTP 也能和 fetch 一样被持久缓存」的挂载点 |
| **3. Full Route Cache（全路由缓存）** | 构建时预渲染的 **RSC Payload / HTML** 等 | **整条路由**在 `next build` 或可静态化时生成产物；动态路由默认更偏运行时 | `/users` 若**强依赖实时 DB**，多为 **dynamic**；此时重点常是第 2 层 + `revalidatePath`，而不是指望整页 build 死缓存 |
| **4. Router Cache（客户端路由缓存）** | 客户端 `<Link>` 等软导航时对 **RSC 响应**的短时内存缓存 | **浏览器侧**；**不能像服务端 `revalidateTag` 那样逐 key 精细清除** | 保存用户后若列表「晚一拍」刷新，要区分：**服务端已失效** vs **客户端仍 holding 旧片段**；有时需 `router.refresh()`（客户端）或接受硬刷新 |

**和常见口述的对齐（避免误解）：**

1. **`unstable_cache` = Data Cache**：给 ORM / SQL 等非 `fetch` 数据源，提供与 `fetch` **同级别的持久化数据缓存**（仍可被 `revalidateTag` / 时间 `revalidate` 管理）。
2. **第 1 层不是 `unstable_cache`**：`cache()` 管的是「**本次请求别重复算**」，不是「下次访问还用服务端 Data Cache 里那份」。
3. **「跨用户、跨刷新」**：Data Cache 条目通常在部署形态下**可被多次请求共享**；具体边界以官方文档与部署环境为准，不要在面试里说成「百分百全局、永不过期」。
4. **第 3 层与 Drizzle**：若路由在构建或某种静态策略下**根本不跑你的数据函数**，则数据库代码不会执行——这是**动态 / 静态**选项问题，与「有没有 `unstable_cache`」是两条轴。

你当前项目为 **Next `16.2.0`**，表格中的 **`'use cache'`** 等能力需以官方配置（如 `cacheComponents`）为准，见下节。

## 1.6 「还有什么 cache？」——和本文强相关的补充

| 名字 | 是什么 | 和 `unstable_cache` 的关系 |
|------|--------|------------------------------|
| **`fetch` + `cache` / `revalidate` / `tags`** | HTTP 响应进入 **Data Cache** | **同一层级**；Drizzle 不走 HTTP 时用 `unstable_cache` 对齐 |
| **React `cache(fn)`** | **请求内** memo | **更浅一层**；常与 `unstable_cache` **组合**（内层去重、外层跨请求），二者**不互斥** |
| **`revalidatePath` / `revalidateTag`** | 按路径或标签**失效缓存** | 主要驱动 **Data Cache** 及关联路由失效；“当前页立刻变不变”常还需要 `refresh()` |
| **`refresh()`** | 刷新 **Router Cache（客户端）** | 常与 `revalidateTag/revalidatePath` 组合：前者保证“服务器下次算新数据”，后者保证“当前操作者立刻看到” |
| **路由段配置**（如 `dynamic`、`revalidate` 导出） | 影响**静态/动态与整段路由行为** | 偏第 3 层与路由语义；不等于给某条 SQL 单独设 key |
| **`'use cache'` 指令**（需按版本开启配置） | **异步组件或异步函数**的输出参与缓存 | 见 **1.7**；偏「输出/片段级」，与 `unstable_cache` 管线不同但可能并存 |

其他容易混进来的词：**`loading.tsx`** 是 Suspense UI，不是 Data Cache；**HTTP `Cache-Control`** 是浏览器/CDN；**客户端 state** 只活在用户浏览器。

## 1.7 怎么「缓存一个组件」？——`export` 和缓存的真实关系

**没有**「给 `export default function Page` 自动套一层全局缓存」的单一语法。实务上通常是：**缓存的是「取数/计算」**，组件仍是正常的 **Server Component `export`**。

### 做法 A：export 普通组件 + 缓存数据函数（最通用、与当前项目最贴）

```tsx
import { cache } from "react";
import { unstable_cache } from "next/cache";

// 第 1 层：同一次请求内去重
export const getUsersOncePerRequest = cache(async () => {
  /* return db.select()... */
});

// 第 2 层：Data Cache（可与 Server Action 里 revalidateTag 联动）
export const getUsersDataCached = unstable_cache(
  async () => {
    /* return db.select()... */
  },
  ["users:list"],
  { tags: ["users"], revalidate: 300 },
);

export default async function UsersPage() {
  const users = await getUsersDataCached();
  return <main>{/* 使用 users */}</main>;
}
```

要点：**`export` 的还是组件**；**持久化**来自 `unstable_cache`（或带缓存的 `fetch`），**请求内去重**可加一层 `cache()`。

### 做法 B：`'use cache'`——组件/函数级「输出」缓存（Next 15+ / 16）

在通过 `next.config` 开启相应能力后，可对 **async** 组件或 **async** 函数写 `'use cache'`，缓存**该段返回值（含 JSX）**，并可用官方配套的 `cacheTag` / `cacheLife` 等（以 [use cache](https://nextjs.org/docs/app/api-reference/directives/use-cache) 为准）。

```tsx
export async function UsersCountBanner(props: { q: string }) {
  "use cache";
  // 示意：内部查库或 fetch；需满足可序列化等约束
  return <p>…</p>;
}
```

**与 `unstable_cache` 的分工（记口诀）：**

- **`unstable_cache`**：包住「async 函数 → 数据」，**到处可复用**（service、工具函数）。
- **`use cache`**：标在 **async 组件或 async 函数**上，偏向「**这一段 UI/输出**」整体进入缓存管线。

项目当前以 **Drizzle service + `revalidatePath`** 为主时，**优先掌握 A + Data Cache** 即可；B 适合在启用 Cache Components 后再系统试错。

## 2. “终极博弈”本质：时间驱动 vs 事件驱动

你可以把缓存更新策略拆成两类：

- **时间驱动**：`revalidate: 60`  
  到点自动过期，简单但不保证“写后立刻一致”
- **事件驱动**：`revalidatePath` / `revalidateTag`  
  在创建、修改、删除后主动清理，能做到“写后尽快一致”

实际项目通常是**混合策略**：

- 读流量大页面：加 `revalidate` 降成本
- 写操作频繁且要求一致：写后调用 `revalidatePath` 或 `revalidateTag`

### 3.1 你现在为什么能更新列表？

## 4. 基于当前项目的升级示例（仅学习示例）

> 以下代码是“你项目可采用的写法示意”，不代表你现在已经启用。

### 4.1 在 `user.service.ts` 给读接口加函数缓存 + tag

```ts
import { unstable_cache } from "next/cache";
import { db } from "@/db";
import { users } from "@/schema";
import { desc } from "drizzle-orm";

const getAllUsersCached = unstable_cache(
  async () => {
    return db.select().from(users).orderBy(desc(users.id));
  },
  ["users:list"], // 缓存键的一部分
  {
    revalidate: 300, // 最长 5 分钟自动过期
    tags: ["users"], // 事件驱动清理入口
  },
);

export async function getAllUsers() {
  return getAllUsersCached();
}
```

### 4.2 在 `user.action.ts` 写后同时清 path 和 tag

```ts
import { revalidatePath, revalidateTag } from "next/cache";

function revalidateUserViews() {
  revalidatePath("/users");
  revalidateTag("users", "fastCache");
}
```

这样做后：

- `revalidatePath("/users")`：保证页面路径相关结果失效
- `revalidateTag("users")`：保证所有打了 `users` tag 的缓存函数一起失效

## 5. `force-cache` 在你项目中的正确落点

你现在 `/users` 读数据来自 DB，不是 HTTP API，所以：

- `force-cache` 不该硬套在 Drizzle 查询上
- 如果未来改成 `fetch("/api/users")` 才考虑：

```ts
await fetch("https://your-domain/api/users", {
  cache: "force-cache",
  next: { tags: ["users"], revalidate: 300 },
});
```

一句话：  
**DB 直查优先 `unstable_cache`；HTTP 请求优先 `fetch + cache/revalidate/tags`。**

## 6. 清除逻辑决策表（面试/实战都好用）

1. **只想按页面失效**：`revalidatePath("/users")`
2. **多页面共用同一批数据**（列表、仪表盘卡片、统计侧栏）：`revalidateTag("users")`
3. **希望自动过期兜底**：给缓存配置 `revalidate: N`
4. **要“写后马上看到新数据”**：写操作后调用 `revalidatePath/revalidateTag`，不要只等时间过期

## 7. 你项目里的一个小提醒

## 9. 一句话总纲（背这个就够）

**四层**：`cache()` / 请求内去重 → **`unstable_cache` / fetch = Data Cache** → Full Route Cache（构建/静态产物）→ Router Cache（客户端软导航）。  

`force-cache` 决定“要不要缓存 fetch”；  
`revalidate` 决定“多久自动过期”；  
`unstable_cache` 解决“非 fetch（如 Drizzle）怎么进 **Data Cache**”；  
`tags + revalidateTag` 解决“按业务事件精准清缓存”；  
`revalidatePath` 解决“按页面路径清缓存”；  
**组件**：通常 **export 普通 Server Component + 缓存数据函数**；进阶用 **`'use cache'`** 标在 async 组件/函数上。
